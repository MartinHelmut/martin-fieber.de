Running tests

With this setup it is now possible to run ./luarocks test. Without any further parameters, this will run a project root test.lua file if found — it will be the entry point for all tests going forward.

For now this file will only print that no tests are defined.

Alternatively, the test property of the project .rockspec file can be used to define a different location or file for the test entry.

Besides using a Lua file as test entry, a shell command can be used by changing script to command.

All options can be found at the LuaRocks GitHub wiki with the test command specification.

Unit testing via LuaUnit

The LuaUnit module installed via LuaRocks is a very popular unit testing framework for Lua. It comes packed with features while still being very easy to set up and use.

Setup LuaUnit

LuaUnit needs to be added to the test dependencies of the project. Dependencies defined under the test_dependencies property in the projects .rockspec file will be automatically installed when running ./luarocks test.

The .rockspec test definition also needs some adjustments, namely running the entry test file with Lua directly to be able to pass command line options. As I really like TAP I will use it as output format for all tests.

Test conventions

For LuaUnit tests are functions or tables of functions that all start with the word test or Test. Every test file needs to at least require luaunit as a module, the root test file then execute those.

A single test

Taking the example function from earlier, a single test case for it can be created by defining a function starting with Test.

Group tests

Multiple test cases should be organised inside a table for grouping. This will increase readability when running the tests.

Multiple files

A growing test suite needs to be organised for better maintainability. Splitting tests in multiple files can be one way. To enable this pattern a test unit needs to return the defined tests, the entry test file will then require all defined tests for execution.

I like keeping tests close to the implementation, that's why all tests for a file are in a file with the same name and the postfix -test before the file extension in the same folder — having a factorial-test.lua test file for the factorial.lua implementation. Mind, this is just a personal preference. Another typical pattern is having a single test folder containing all test files.

Now with the dedicated test files, the entry file will require all tests and run LuaUnit.

Running ./luarocks test will then execute the test suite.

Mocking

There are many ways to isolate code and replace external dependencies for testing, called mocking. In my time using Lua there was sadly no test-mock library that won my heart, instead I use a combination of different techniques to mock code for testing.

Testability of code

Increasing testability should always be the first step, avoiding any work-around to mock code altogether. Instead of accessing a dependency inside a function, passing it as argument; initialising an object with needed dependencies; or making default values accessible to override via function or property.

Basic mock

A basic mock can be an empty structure that replaces a dependency for a different chunk of code. What this structure does or how it is shaped depends on the goal of the test — e.g. an empty function can be passed to skip or remove a call, checking if a method actually got called, or simulating values on a table to fully run a function.

To showcase this, the following createUser function will be tested, the callback mocked.

The goal here is to test if the user gets created successfully, therefore an empty callback function is of no use. Only if the callback gets called the user got created.

One way to test this is to create a mock function that toggles a boolean when called, and asserting the value.

Mock exported functions and values

Mocking a function exported by a module is best done inside LuaUnit's setUp and tearDown functions. The exported function or value gets replaced with a mock in setUp, keeping a reference, and restoring the original value in tearDown.

Mock globals

As Lua keeps all global variables in a table called _G, the environment, global functions and values can be mocked for tests the same way as exported values.
